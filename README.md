## Описание

Скрипт проверяет актуальность версий всех компонентов Pangolin-стека, сравнивает их с установленными и при наличии обновлений выполняет полный цикл обновления. После запуска новых контейнеров производится многоступенчатая проверка работоспособности. Если стек не прошёл проверку — скрипт автоматически откатывает конфигурацию и поднимает предыдущие версии из локального кэша образов.

**Порядок выполнения при наличии обновлений:**

1. Создание резервных копий конфигурационных файлов
2. Остановка Docker-стека
3. Обновление версий в `docker-compose.yml` и `traefik_config.yml`
4. Скачивание новых Docker-образов
5. Запуск стека на новых версиях
6. **Health check** — проверка состояния контейнеров (с повторными попытками)
7. При успехе — удаление устаревших образов
8. При сбое — **автоматический откат** к предыдущим версиям

> Удаление старых образов (`docker image prune`) происходит **только после успешного** прохождения health check, что гарантирует возможность отката.

## Требования

- **OS:** Linux (Bash 4.0+)
- **Права:** root (или пользователь с доступом к Docker)
- **Зависимости:**

| Утилита | Назначение |
|---------|-----------|
| `curl` | Запросы к GitHub API и Docker Hub |
| `jq` | Парсинг JSON-ответов |
| `docker` | Управление контейнерами (`compose`, `inspect`, `logs`) |
| `sed` | Замена версий в конфигурационных файлах |
| `awk` | Извлечение версии Badger из YAML |

Установка зависимостей на Debian/Ubuntu:

```bash
apt install curl jq docker.io
```

## Конфигурация

Все настройки сосредоточены в блоке **«Конфигурация»** в начале скрипта:

```bash
# Пути
PANGOLIN_DIR="/root/pangolin"
COMPOSE_FILE="${PANGOLIN_DIR}/docker-compose.yml"
TRAEFIK_CONFIG="${PANGOLIN_DIR}/config/traefik/traefik_config.yml"
LOG_FILE="/var/log/pangolin-update.log"
LOCK_FILE="/tmp/pangolin-update.lock"
LOG_MAX_LINES=1000       # ротация лога при превышении этого числа строк

# Параметры health check
HEALTH_INITIAL_WAIT=20   # пауза после запуска перед первой проверкой (сек)
HEALTH_INTERVAL=10       # интервал между повторными проверками (сек)
HEALTH_RETRIES=6         # максимальное количество попыток
RESTART_THRESHOLD=3      # порог перезапусков контейнера, считается аномалией
```

Суммарное время ожидания health check: `HEALTH_INITIAL_WAIT + HEALTH_INTERVAL × HEALTH_RETRIES` = **80 секунд** при настройках по умолчанию.

## Логика работы

```
Старт
  │
  ├─► Ротация лога            (если > LOG_MAX_LINES строк)
  ├─► Проверка зависимостей   (curl, jq, docker, sed, awk)
  ├─► Захват lock-файла       (защита от параллельного запуска)
  ├─► Чтение текущих версий   (из docker-compose.yml и traefik_config.yml)
  ├─► Запрос актуальных версий (GitHub API + Docker Hub API)
  ├─► Вывод таблицы сравнения
  │
  ├─[Все версии совпадают]──► Выход (код 0)
  │
  └─[Есть обновления]
        │
        ├─► Создание резервных копий конфигов  (.bak.DDMMYYYY_HHMMSS)
        ├─► docker compose down
        ├─► sed: замена версий в конфигурационных файлах
        ├─► docker compose pull
        ├─► docker compose up -d
        │
        └─► Health check (с повторами)
              │
              ├─[Успех]──► docker image prune ──► Выход (код 0)
              │
              └─[Сбой]
                    │
                    ├─► dump_failed_logs  (логи упавших контейнеров → LOG_FILE)
                    │
                    └─► Откат
                          ├─► docker compose down
                          ├─► Восстановление конфигов из резервных копий
                          ├─► docker compose up -d  (старые образы в кэше)
                          ├─► Проверка состояния после отката
                          │
                          ├─[Откат успешен]──► Выход (код 1)
                          └─[Откат провален]──► Выход (код 2)  ⚠ ручное вмешательство
```

## Health check

После запуска стека скрипт проверяет каждый контейнер через `docker inspect` по трём критериям:

| Критерий | Условие прохождения |
|----------|-------------------|
| `State.Status` | Значение равно `running` |
| `RestartCount` | Меньше `RESTART_THRESHOLD` (по умолчанию 3) |
| `State.Health.Status` | `healthy` или `none` (healthcheck не определён) |

Если хотя бы один контейнер не удовлетворяет критериям — попытка считается неудачной. Скрипт повторяет проверку через `HEALTH_INTERVAL` секунд, всего до `HEALTH_RETRIES` раз.

### Временна́я шкала health check (значения по умолчанию)

```
Запуск контейнеров
       │
    +20с  (HEALTH_INITIAL_WAIT)
       │
   Попытка 1 ──[OK]──► Готово
       │
    +10с  (HEALTH_INTERVAL)
       │
   Попытка 2 ──[OK]──► Готово
       │
    ...до 6 попыток (HEALTH_RETRIES)
       │
   Провал ──► Откат
```

Максимальное суммарное время ожидания: **80 секунд**.

### Диагностика при сбое

При провале health check скрипт автоматически собирает последние 50 строк логов каждого проблемного контейнера и записывает их в `LOG_FILE`. Это позволяет разобраться в причине сбоя без дополнительных команд.

## Автоматический откат

Откат запускается автоматически, если health check провален после всех попыток.

### Механизм отката

Старые образы Docker **не удаляются** до успешного прохождения health check, поэтому они гарантированно доступны в локальном кэше для немедленного перезапуска без скачивания.

**Шаги отката:**

1. `docker compose down` — остановка контейнеров с новыми версиями
2. Восстановление `docker-compose.yml` из резервной копии `.bak.ДАТА`
3. Восстановление `traefik_config.yml` из резервной копии `.bak.ДАТА`
4. `docker compose up -d` — запуск на старых образах из кэша
5. Проверка состояния контейнеров после отката

### Ручное восстановление

Если автоматический откат по какой-то причине тоже завершился неудачно (код выхода `2`), восстановить стек можно вручную:

```bash
cd /root/pangolin

# Найти резервные копии
ls -lt docker-compose.yml.bak.*

# Восстановить конфиги (заменить суффикс на актуальный)
cp docker-compose.yml.bak.14042026_030005 docker-compose.yml
cp config/traefik/traefik_config.yml.bak.14042026_030005 config/traefik/traefik_config.yml

# Поднять стек
docker compose up -d

# Убедиться, что всё запустилось
docker compose ps
```

## Логирование

Лог записывается в `/var/log/pangolin-update.log`.

### Формат записи

```
[YYYY-MM-DD HH:MM:SS] [LEVEL] Сообщение
```

### Уровни

| Уровень | Цвет в терминале | Описание |
|---------|-----------------|----------|
| `INFO` | Голубой | Информационные сообщения о ходе работы |
| `OK` | Зелёный | Успешное завершение шага |
| `WARN` | Жёлтый | Предупреждения и сообщения об откате |
| `ERROR` | Красный (stderr) | Критические ошибки |

> Цвета выводятся только в интерактивном терминале (TTY). При запуске через cron лог пишется без escape-последовательностей.

### Ротация лога

При превышении `LOG_MAX_LINES` (по умолчанию 1000 строк) текущий лог переименовывается:

```
/var/log/pangolin-update.log  →  /var/log/pangolin-update.log.YYYYMMDD
```

Старые файлы не удаляются автоматически — при необходимости настройте `logrotate`.

### Просмотр лога

```bash
# Последние 50 строк
tail -50 /var/log/pangolin-update.log

# Только ошибки
grep '\[ERROR\]' /var/log/pangolin-update.log

# Только откаты
grep 'ОТКАТ\|Откат' /var/log/pangolin-update.log

# История успешных обновлений
grep 'Обновление успешно завершено' /var/log/pangolin-update.log

# Провалы health check
grep 'Health check провален' /var/log/pangolin-update.log
```

## Резервные копии

Перед каждым обновлением создаются копии конфигурационных файлов:

```
docker-compose.yml                →  docker-compose.yml.bak.14042026_030005
config/traefik/traefik_config.yml →  config/traefik/traefik_config.yml.bak.14042026_030005
```

Суффикс включает дату и время (`DDMMYYYY_HHMMSS`), что исключает коллизии при нескольких запусках в день.

## Автоматизация через cron

Для ежедневной проверки обновлений в 3:00 ночи:

```bash
crontab -e
```

```cron
0 3 * * * /usr/local/bin/pangolin-update.sh
```

Для еженедельной проверки (в воскресенье в 4:00):

```cron
0 4 * * 0 /usr/local/bin/pangolin-update.sh
```

> При запуске через cron цвета в логе отключаются автоматически (нет TTY). Все ключевые сообщения пишутся в `LOG_FILE`.

---

## Коды выхода

| Код | Ситуация | Действие |
|-----|----------|----------|
| `0` | Успех: обновление выполнено или не требуется | — |
| `1` | Health check провален, откат выполнен успешно | Стек работает на предыдущих версиях. Проверьте лог для диагностики. |
| `2` | Health check провален, откат тоже провален | **Требуется ручное вмешательство.** Используйте резервные копии `*.bak.ДАТА`. |

## Структура файлов

```
/root/pangolin/
├── docker-compose.yml                          # Конфиг стека (обновляется скриптом)
├── docker-compose.yml.bak.DDMMYYYY_HHMMSS      # Резервная копия
└── config/
    └── traefik/
        ├── traefik_config.yml                  # Конфиг Traefik/Badger (обновляется скриптом)
        └── traefik_config.yml.bak.DDMMYYYY_HHMMSS  # Резервная копия

/var/log/
├── pangolin-update.log                         # Текущий лог
└── pangolin-update.log.YYYYMMDD                # Архивный лог (после ротации)

/tmp/
└── pangolin-update.lock                        # Lock-файл (удаляется при завершении)

/usr/local/bin/
└── pangolin-update.sh                          # Скрипт
```



## Установка и Обновление newt

Создание файла службы:

```bash
sudo nano /etc/systemd/system/newt.service
```

Содержимое файла:

```bash
[Unit]
Description=Newt Agent
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/newt --id 9***on --secret py5***qj66 --endpoint https://domain.tld
Restart=always
RestartSec=5
User=root
WorkingDirectory=/usr/local/bin

# Optional hardening
LimitNOFILE=65536
NoNewPrivileges=true

[Install]
WantedBy=multi-user.target

systemctl daemon-reload
```
Обновите конфигурации служб

```bash
systemctl daemon-reload
```

Для обновления сначала необходимо остановить службу: `systemctl stop newt.service`

Загрузите новый бинарный файл newt^

```bash
NEWT_V=$(curl -s https://api.github.com/repos/fosrl/newt/releases/latest | grep '"tag_name"' | cut -d '"' -f 4) \
  && [ -n "$NEWT_V" ] \
  && wget "https://github.com/fosrl/newt/releases/download/$NEWT_V/newt_linux_amd64" -O /usr/local/bin/newt \
  && chmod +x /usr/local/bin/newt
```

Запустите службу: `systemctl start newt.service`
